# persona-audit

[English](README.md) · [中文](README.zh.md) · **日本語**

**4 人の架空ユーザーが、あなたのツールが本物のユーザーに見せる文章を読み、どこが意味不明かを教えてくれる。**

自分のプロダクトに近すぎて、初見の人がどこでつまずくか見えなくなっていませんか。persona-audit は 4 つのペルソナを立ち上げ、コードもドキュメントも見ずに、ツールの**出力テキストだけ**を読みます。そして誤読した箇所・不安になった箇所・見つけられなかった箇所を報告します。**合意マトリクス**が、複数のペルソナが独立して引っかかった点を浮かび上がらせる。そこが最初に直すべき場所です。

![persona-audit を自身の README にかけた結果——4 ペルソナ、5 つの指摘](docs/hero-consensus-matrix.ja.png)

> 上のスクリーンショットは、persona-audit をこの README 自身にかけた結果です。「自分では見えない盲点を見つける」と謳うツールの README に、4 つのペルソナが揃って一番大きな盲点を指摘しました。出力例が一つも載っていない、と。だから今は載せています。この一枚です。

## 実際のプロダクトでの例

たとえば、アプリがユーザーに毎日送る要約メールに向けてみます。4 つのペルソナが、同じ実物の要約をコールドリードします：

- **初心者：**「『今週：0』と出ていて、アプリが壊れたのかと思った」
- **熟練者：**「『純増減』と『合計』が両方あって、どっちがどっちか分からない」
- **ターゲットユーザー：**「明日また開く理由がどこにも書かれていない」

3 人の違う人が、同じ一点で混乱する → 合意マトリクスがそれを最上位に並べ、ship する前にその一文を直せる。（完全なサンプル実行：[`examples/case-study.md`](examples/case-study.md)）

## 使うとき

✅ **エンジンがユーザー向けテキストを生成する**ツールが、新バージョン／新しい出力を ship したとき——レポート生成、デイリーダイジェスト、bot のプッシュ文面、読み上げ系ツール。
❌ コードレビュー · 純粋な UI／ビジュアル QA · インタラクティブ・ナラティブの QA。

## 動作の仕組み（と用語の意味）

1. **実際の出力を 6〜8 本生成する**——通常ケース、エッジデータ、空の状態、そしてユーザーが実際に見るビューをカバー。
2. **4 ペルソナがコールドリード**——*コールドリード* = コードを見ず出力だけを見て、初見のユーザーのように反応する。4 つは：ミラー（あなたの実ユーザー）、初心者（専門用語＋不安の検出器）、熟練者（見せかけの精度の検出器）、新機能のターゲットユーザー。
3. **合意マトリクス**——複数のペルソナが独立して同じ点を突いた数。単発の重箱の隅ではなく、複数の角度からの一致こそがシグナル。
4. **ABCD トリアージ**——**A** すぐ直す（誤字、数字のズレ）· **B** 作るべき本当のギャップ · **C** 意図した設計判断と衝突する → *触る前にオーナーに確認* · **D** 範囲外。
5. **削ってはいけないリスト**——ユーザーが実際に気に入っているものを記録し、次のバージョンでうっかり削らないようにする。

## 「synthetic users」との違い

ああいうツールはユーザーをシミュレートして**リサーチを行い**、LLM の出力を「測定」に見せかけがちです。persona-audit は逆です。**すでに出している**テキストを監査し、出力は検証すべき候補リストであって、データではないと正直に言います。

## これが何で、何でないか（正直に）

一人で、追加コストなし、数分でできる**盲点スキャン**であって、校正済みの測定器ではありません。4 つのペルソナは同一のベースモデルを共有するため、「1 票」は*複数の角度が一致した*という意味で、統計的独立ではありません。複数角度の一致に従って直し、確定として扱う前に検証してください。
