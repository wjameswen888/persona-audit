# persona-audit

[English](README.md) · [中文](README.zh.md) · **日本語**

**4 人の架空ユーザーが、あなたのツールが本物のユーザーに見せる文章を読み、どこが意味不明かを教えてくれる。**

自分のプロダクトに近すぎて、初見の人がどこでつまずくか見えなくなっていませんか。persona-audit は 4 つのペルソナを立ち上げ、コードもドキュメントも見ずに、ツールの**出力テキストだけ**を読みます。そして誤読した箇所・不安になった箇所・見つけられなかった箇所を報告します。

## 何を見つけるか

たとえば、アプリがユーザーに毎日送る要約メールに向けてみます。4 つのペルソナが、同じ実物の要約を*コールドリード*します——*コールドリード* = コードを見ず出力だけを見て、初見のユーザーのように反応すること：

- **初心者：**「『今週：0』と出ていて、アプリが壊れたのかと思った」
- **熟練者：**「『純増減』と『合計』が両方あって、どっちがどっちか分からない」
- **ターゲットユーザー：**「明日また開く理由がどこにも書かれていない」

3 つの違う角度が、同じ一点で混乱する。**合意マトリクス**——複数のペルソナが独立して引っかかった数で並べたリスト——がその一文を最上位に押し上げ、ship する前に直せる。単発の重箱の隅ではなく、複数角度の一致こそが、従うべきシグナルです。

![persona-audit が生成する合意マトリクス——複数のペルソナが独立して引っかかった数で並んでいる](docs/hero-consensus-matrix.ja.png)

> 上のスクリーンショットは、persona-audit をこの README 自身にかけた結果です。「自分では見えない盲点を見つける」と謳うツールに、出力例が一つも載っていないと 4 つのペルソナが揃って指摘しました。だから今は載っています——あなたが今読んだあのブロックです。

## Claude にレビューを頼むのと何が違う？

「これをレビューして」という普通のプロンプトは、あなたの言い回しに引きずられ、聞きたい答えを返しがちです。persona-audit は 4 人の他人を固定の人物像に縛りつけ、誰にもコードを見せず、*複数のペルソナが独立して引っかかった*ものだけを掬い上げます——従順な一回のレビューがそのまま素通りする盲点を、あぶり出します。

## 使うとき

✅ **エンジンがユーザー向けテキストを生成する**ツールが、新バージョン／新しい出力を ship したとき——レポート生成、デイリーダイジェスト、bot のプッシュ文面、読み上げ系ツール。*（そう、ユーザーに毎日まとめを送る個人プロジェクトも対象です。）*
❌ コードレビュー · 純粋な UI／ビジュアル QA · インタラクティブ・ナラティブの QA。

## 動作の仕組み

1. **実際の出力を 6〜8 本生成する**——通常ケース、エッジデータ、空の状態、そしてユーザーが実際に見るビューをカバー。
2. **4 ペルソナがコールドリード**——ミラー（あなたの実ユーザー）、初心者（専門用語＋不安の検出器）、熟練者（見せかけの精度の検出器）、そして今 ship したものに対するターゲットユーザー。
3. **合意マトリクス**——複数のペルソナが独立して同じ点を突いた数で並べる。「1 票」は複数の角度が一致したという意味——「まず検証すべき」という強いヒントであって、証拠ではありません。
4. **ABCD トリアージ**——**A** すぐ直す（誤字、数字のズレ）· **B** 作るべき本当のギャップ · **C** 意図した設計判断と衝突する → *触る前にオーナーに確認* · **D** 範囲外。
5. **削ってはいけないリスト**——ユーザーが実際に気に入っているものを記録し、次のバージョンでうっかり削らないようにする。

## これが何で、何でないか——そして「synthetic users」との違い（正直に）

ああいうツールはユーザーをシミュレートして**リサーチを行い**、LLM の出力を「測定」に見せかけがちです。persona-audit は逆です。**すでに出している**テキストを監査し、自らの限界をはっきり認めます。4 つのペルソナは同一のベースモデルを共有するため、「1 票」は*複数の角度が一致した*という意味で、統計的独立ではありません——これは一人で、追加コストなし、数分でできる、**検証すべき候補リストを返す盲点スキャンであって、データではありません**。複数角度の一致に従って直し、確定として扱う前に検証してください。
