# persona-audit

[English](README.md) · **中文** · [日本語](README.ja.md)

**四个假用户，读你给真用户看的文字，告诉你哪句不通。**

你太懂自己的产品，看不见陌生人卡在哪。persona-audit 派四个画像，**只读**你工具输出的文字——不碰代码、不看文档——报告他们读错了什么、被什么吓到、找不到什么。一张**共识矩阵**显示几个画像各自独立撞上同一个问题。那就是你最该先改的。

![persona-audit 跑在自己 README 上——四个画像，五条发现](docs/hero-consensus-matrix.zh.png)

> 上面这张图，就是 persona-audit 跑在这份 README 上的结果。四个画像各自独立地指出：一个讲"帮你看见自己盲区"的工具，README 最大的盲区是——它没展示任何 demo。所以现在有了，就是这张。

## 在真实产品上长什么样

拿你 app 每天发给用户的那条简报举例。四个画像冷读同一条真实简报：

- **新手：**"上面写'本周：0'，我以为 app 坏了。"
- **老手：**"'净变化'和'总计'都在，我分不清哪个是哪个。"
- **目标用户：**"没有一句告诉我明天为什么还要点开它。"

三个不同的人，撞上同一处困惑 → 共识矩阵把它排到最前，你在 ship 之前就把这句改掉。（完整示例：[`examples/case-study.md`](examples/case-study.md)。）

## 什么时候用

✅ 一个**引擎产出用户向文本**的工具 ship 了新版本/新输出——报告生成器、每日简报、bot 推送文案、读数工具。
❌ 代码 review · 纯 UI/视觉 QA · 交互叙事 QA。

## 怎么跑（以及这些词什么意思）

1. **生成 6–8 份真实输出**——覆盖普通情况、边缘数据、空状态、用户真正看到的那个视图。
2. **四个画像冷读**——*冷读* = 只看输出、不看代码，像真用户第一次见那样反应。四个：镜像（你的真实用户）、新手（黑话+恐慌探测器）、老手（假精度探测器）、新功能目标用户。
3. **共识矩阵**——几个画像各自独立撞上同一处。跨角度的一致、而不是单点挑刺，才是信号。
4. **ABCD 分层**——**A** 立刻修（错别字、数字错位）· **B** 要做的真 gap · **C** 跟某个刻意的设计选择冲突 → *动手前先问 owner* · **D** 边界外。
5. **别砍清单**——用户真正喜欢的东西，标出来，免得下版本顺手砍了。

## 跟"合成用户"有什么不一样

那些工具模拟用户去**做调研**，还常把 LLM 输出包装成"测量"。persona-audit 反过来：它审你**已经在发**的文字，而且坦白产出是一份待核实的候选清单——不是数据。

## 诚实说它是什么

一个单人、零成本、几分钟的**盲区扫描**——不是校准过的测量。四个画像共用同一个底层模型，所以"一票"意味着*几个角度都觉得别扭*，不是统计独立。照着跨角度的一致去改；当作结论前先核实。

## 安装

```bash
git clone https://github.com/wjameswen888/persona-audit.git
cp -r persona-audit ~/.claude/skills/
```

这是个 [Claude Code skill](https://docs.claude.com/en/docs/claude-code)——按描述里的触发词自动触发。把 `LOCAL.md.example` 复制成 `LOCAL.md` 来绑定你的产品（已 gitignore；每个钩子都有默认值，不填也能跑）。

## 文件

| 文件 | 是什么 |
|------|--------|
| `SKILL.md` | 方法本体——通用、可移植 |

| `examples/case-study.md` | 一场从头到尾的真实运行 |
| `LOCAL.md.example` | 你私有产品绑定的模板 |
